# 通知（完了 / QA）仕様

> エージェントの**完了（レビュー待ち）**や**QA発生**を、ブラウザを通じてユーザーに通知する機能の仕様。
> ステータス: ドラフト / 最終更新: 2026-06-05

---

## 1. 目的・スコープ
- 工程の Run が **人間の対応を要する状態**（QA待ち / レビュー待ち 等）になったら、ユーザーに気づかせる。
- **どのワークフローを開いていても**通知が届く（クロスワークフロー）。
- 各ワークフローに **要対応件数のバッジ**を表示し、未対応の仕事が一目で分かる。
- 対象: アプリのタブが開いている間（バックグラウンドタブ可）。**タブを完全に閉じた場合は対象外**（後述「制約」）。

---

## 2. 主要要件（今回の確認事項）
1. **クロスワークフロー通知**: 別ワークフロー閲覧中でも、全ワークフロー・全Runのイベントで通知が出る。
2. **ワークフロー別バッジ**: トップバーのワークフロー切替（セレクト/リスト）に、**ワークフローごとの要対応件数**を表示する。

---

## 3. 配信アーキテクチャ（グローバルイベントWS）

現状の WebSocket は `/ws/runs/{run_id}` で**選択中Run専用**。これだけでは「別ワークフローの裏で起きた変化」を拾えない（アイドル時はポーリングも止まる）。そこで：

- **バックエンド**: `EventHub` に**グローバル購読**を追加し、`publish()` 時に**全イベントをグローバルチャンネルへも配信**（`run_id` / `process_id` / `workflow_id` を付与）。
- 新ルート **`GET /ws/events`**（WebSocket）: クライアントが1本だけ接続し、全Runの状態変化を受信。
- フロントは**アプリ起動時に1本だけ**接続し、自動再接続。選択中Runの細粒度WS（ログ/usage）はこれまで通り併用。

> これにより「他のワークフローを参照中でも通知が来る」を満たす。

---

## 4. 通知トリガー（イベント種別）
| イベント | 状態 | 既定 | 文言例 |
|---|---|---|---|
| **QA発生** | `waiting_qa` | **ON** | 「QA: 〈工程名〉が回答待ちです」 |
| **レビュー待ち** | `in_review` | **ON** | 「レビュー: 〈工程名〉が提出されました」 |
| **失敗** | `failed` | **ON** | 「失敗: 〈工程名〉が中断しました」 |
| **承認/差し戻し確定** | `approved` / `rejected` | OFF（任意） | 「〈工程名〉が承認/差し戻しされました」 |

- **【確定】** 既定は **QA待ち / レビュー待ち / 失敗** の3つ。`approved`/`rejected` は任意（既定OFF）。
- 設定は `runtime_settings.json`（既存のアプリ設定）に `notify_events: ["waiting_qa","in_review","failed"]` として保存。

---

## 5. ブラウザ通知（Web Notifications API）
- **権限取得**: トップバーの**ベル・トグル**押下で `Notification.requestPermission()`（ユーザー操作が必須）。可否は localStorage に保存。localhost は通知要件（HTTPS or localhost）を満たす。
- **発火**: グローバルWSで対象イベント受信時に `new Notification(title, { body, tag, data })`。
  - `tag` に `run_id` を使い、同一Runの通知は**上書き**（連続発火の氾濫防止）。
  - **クリック動作**: `window.focus()` ＋ 該当ワークフローへ切替＋該当Runを選択（`data` に `workflow_id`/`run_id`）。
- **抑止**: 既にそのRunを画面で見ている（フォアグラウンド＋選択中）ときは通知を出さない。
- **重複防止**: `run_id × status` 単位で last-notified を記録し、同じ遷移の二重通知を防ぐ。
- **フォールバック**: 権限が無い/拒否のときは**アプリ内トースト**で代替（任意で通知音）。

---

## 6. ワークフロー別バッジ（要対応件数）

### 6.1 【確定】現状態からの**派生カウント**（ストア不要）
- バッジの数 = そのワークフロー内で、**各工程の最新Runが「要対応状態」**であるものの数。
  - 要対応状態 = `waiting_qa` ＋ `in_review` ＋ `failed`（設定の `notify_events` と連動）。
- **自動クリア**: QAに回答 → `running`、レビュー承認 → `approved`、失敗を resume/再実行 → 新しい最新Run、等で**状態が変われば自然に件数から外れる**。**リロード/複数タブでも常に正確**（永続フラグ不要）。
- 「未読」ではなく「**未対応の実数**」を表す。**未読ストア方式は採用しない（不要）。**

### 6.2 【不採用】通知ストア（未読フラグ）
- 「前回見てから来た未読」セマンティクスは**今回は不要**のため実装しない。将来必要になった場合のみ `notification` テーブル方式を検討。

### 6.3 表示: **種別アイコン**【確定】
- ワークフロー切替の各項目に、**種別ごとのアイコン＋件数**を並べて表示（合計だけの単一バッジにはしない）。
  - 例: 🗨️`1`（QA待ち） ✅`1`（レビュー待ち） ⚠️`2`（失敗）
- アイコン対応（lucide想定）: QA待ち=`MessageSquare`、レビュー待ち=`Inbox`（または`Check`）、失敗=`AlertTriangle`。
- 件数 0 の種別はアイコン非表示。全種別 0 ならバッジ無し。
- 色分け: QA=青系 / レビュー=緑系 / 失敗=赤系。ツールチップに状態名と件数。

## 7. データ / API

### 7.1 グローバルWS ペイロード（`/ws/events`）
```json
{ "type": "status", "run_id": "run_...", "process_id": "proc_...",
  "workflow_id": "wf_...", "payload": { "status": "in_review" } }
{ "type": "qa", "run_id": "run_...", "process_id": "proc_...",
  "workflow_id": "wf_...", "payload": { "id": "qa_...", "question_text": "..." } }
```

### 7.2 要対応サマリー（バッジ初期化・再同期用）
- **`GET /api/attention`** → 各ワークフローの**種別別**要対応件数。
```json
[ { "workflow_id": "wf_1", "waiting_qa": 1, "in_review": 1, "failed": 2 } ]
```
  - 算出: ワークフロー内の各工程の**最新Run**の status を集計（`waiting_qa` / `in_review` / `failed`）。
  - フロントはこの3種をそのまま種別アイコン（§6.3）に割り当てる。
- フロントは起動時にこれでバッジ初期化し、以後はグローバルWSの増減で更新（必要なら再フェッチ）。

### 7.3 設定（既存 `runtime_settings` を拡張）
- `notify_events: string[]`（通知/バッジ対象の状態）。
- `notify_enabled: bool`（ブラウザ通知のON/OFF。権限とは別の意思表示）。

---

## 8. UI
- **トップバー**: ベル・トグル（通知の有効化/権限要求、状態をアイコンで表示）。
- **ワークフロー切替**: 各項目に**種別アイコン＋件数**（🗨️QA / ✅レビュー / ⚠️失敗、0は非表示。§6.3）。
- **通知クリック**: 当該ワークフローへ切替→Run選択→QA/レビューUIへスクロール。
- フォールバックのアプリ内トースト（権限なし時）。

---

## 9. 制約（正直に）
- 通知はタブが**開いている間**のみ（バックグラウンドタブ可）。**タブを閉じると JS が動かず通知不可**。
- タブを閉じても通知したい場合は **Service Worker + Push API + プッシュサーバ**が必要。ローカル単一ユーザ用途には過剰なため**今回は対象外**を推奨。
- 通知のOS表示は各ブラウザ/OSの設定にも依存（サイレント時間帯等）。

---

## 10. 実装インパクト（参考）
- **backend**: `events.py`（グローバル購読＋`publish` で全体配信）、`api.py`（`/ws/events`, `GET /api/attention`）、状態遷移時に `workflow_id` を引けるよう `_publish` 呼び出しへ付与。6.2採用時のみ `notification` テーブル＋関連API。
- **frontend**: `useNotifications` フック（権限・グローバルWS・desktop通知・重複抑止）、`useAttention`（バッジ件数）、トップバーのベル、ワークフロー切替のバッジ。設定への `notify_events`/`notify_enabled`。
- **スキーマ変更**: 6.1 推奨構成なら**不要**（派生集計のみ）。6.2 採用時のみ `notification` テーブル追加。

---

## 11. 確定事項
- **通知対象**: `waiting_qa` / `in_review` / `failed` の3種（`approved`/`rejected` は任意・既定OFF）。
- **バッジ**: 現状態からの**派生カウント（未対応の実数, §6.1）**。未読ストア（6.2）は**不採用**。スキーマ変更不要。
- **バッジ表示**: **種別アイコン＋件数**（QA / レビュー / 失敗、§6.3）。
